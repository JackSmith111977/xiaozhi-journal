# Epic 9 Retrospective: 数据云端同步

**Date:** 2026-04-19
**Epic Status:** done (5/5 stories completed)
**Stories:** 9-1 (数据库迁移 + RLS), 9-2 (Realtime 订阅), 9-3 (离线同步), 9-4 (数据导出), 9-5 (账户删除)

## Summary

Epic 9 完成了从本地应用到云端同步的核心转型：6 张 Supabase 表 + RLS 策略、实时订阅、离线-first 架构、GDPR 合规（数据导出 + 账户删除）。所有 Story 通过 TypeScript 编译验证和 code review。

## What Went Well

1. **数据库先行策略** — Story 9.1 率先完成迁移和 RLS，为后续所有 story 提供了可靠的数据层基座
2. **数据库 trigger 替代客户端逻辑** — handle_new_user() trigger 替代了客户端 profile 创建，消除了竞态条件
3. **错误处理一致性** — Story 9.3 将 syncToSupabase 从 `return` 改为 `throw`，确立了错误传播的一致性模式
4. **资源泄漏修复能力** — 跨 story 的 event listener 和 setTimeout 泄漏被系统性发现并修复
5. **GDPR 合规一步到位** — 数据导出（9.4）和账户删除（9.5）同时满足数据携带权和被遗忘权

## What Didn't Go Well

1. **Story 9.4/9.5 review 记录缺失** — 首次实现时 Dev Agent Record 和 Senior Developer Review 为空或极简短，review 流程被跳过
2. **基础设施依赖跨 epic** — profiles 表和 avatars bucket 属于 9.1 但 8.4 需要它们，导致 8.4 的 profile 管理功能无法在本地完整验证
3. **内存泄漏模式反复出现** — 9.3 的 event listener 泄漏、9.5 的 setTimeout 泄漏，反映了对异步资源生命周期的注意力不够
4. **DB ↔ TS 类型鸿沟** — status 值在 DB（draft/published/archived）和 TS（pending/ai_ready/ai_done）之间不一致，需要 statusMap 映射层
5. **Auth 用户删除未实现** — 账户删除只删除了 profile + signOut，auth.users 中的记录依然存在，用户重新注册相同邮箱时会冲突

## Technical Debt Carried Forward

- **Auth 用户未删除** — 账户删除后 auth.users 记录残留，需 Edge Function 清理
- **30 天备份清除** — Supabase 不会自动清除 30 天内的备份中的用户数据，需手动操作或 Edge Function
- **pendingMessage 不自动清除** — 离线提示消息不会自动消失，影响用户体验
- **journals 缺少 (user_id, updated_at DESC) 索引** — 影响 last-write-wins 查询性能

## Key Lessons

1. **基础设施依赖应前置** — 任何 story 需要的数据库表/Storage bucket 应在 Epic 开始前识别并完成
2. **资源泄漏是隐蔽的 bug** — 在 SPA 中，每次 addEventListener 和 setTimeout 都必须有对应的 remove/cleanup
3. **类型一致性胜过局部优化** — DB 和 TS 之间使用不同枚举值看似灵活，实则增加维护成本和出错概率
