# Epic 8 Retrospective: 用户注册与登录

**Date:** 2026-04-19
**Epic Status:** done (4/4 stories completed)
**Stories:** 8-1 (邮箱注册), 8-2 (邮箱登录), 8-3 (密码重置), 8-4 (个人资料管理)

## Summary

Epic 8 建立了完整的用户认证体系：注册 → 登录 → 密码重置 → 个人资料管理。所有 Story 通过 TypeScript 编译验证，代码质量在多轮 review 中持续提升。

## What Went Well

1. **代码质量逐轮提升** — 6+ 轮 code review，P0 问题从首轮多发到后续基本清零，体现了 review 流程的有效性
2. **Graceful Fallback 模式** — profiles 表不存在时降级显示邮箱，为后续依赖未完成基础设施的场景建立了可复用模式
3. **认证架构模式稳固** — AuthGuard 客户端守卫 + Zustand store + Supabase session 三方同步，在 4 个 Story 中保持一致
4. **安全意识到位** — 统一错误信息防止邮箱枚举、密码重置 token 服务端验证、密码强度校验

## What Didn't Go Well

1. **基础设施依赖未前置** — profiles 表和 avatars bucket 属于 9.1 的任务，但 8.4 依赖它们才能实际验证功能。应在 Epic 开始前识别并前置这些依赖
2. **Review diff 误判** — 部分 review agent 报告的 P0 问题（如 `e.token.value`、缺少闭合括号）在实际文件中不存在，是 unified diff 展示造成的视觉误导
3. **密码重置 redirect URL 的 SSR 安全问题** — 最初在共享 lib 中使用 `window.location.origin` 导致 SSR 崩溃，后改为参数传入，但增加了调用方的负担
