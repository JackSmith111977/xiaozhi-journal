## 产品故事

### 你为什么写日记？

不是因为"应该记录"，而是因为心里有东西在转。

写不出来、写不下去、写了也不回头看——这是所有日记 App 都解决不了的问题。不是工具不好用，是因为**一个人的时候，最难被看见的是自己**。

### 如果有个人，真的读懂了你

不是分析你的情绪是 73% 焦虑，而是说：**"你今天写得很短很急，是在赶时间吗？"**

不是告诉你"加油，你可以的"，而是说：**"改了 5 遍还在改，说明你没放弃自己。"**

不是给你一堆数据和图表，而是给你一句说到心坎里的话：**"第 5 版不是失败，是第 5 次不肯妥协的自己。"**

这个人叫**小知**——一个温暖、有个性、永远不评判你的朋友。

### 用户第一次被打动的瞬间

1. 注册/登录后，看到一条属于自己的情绪波形图
2. 点"😊"，写了一句"今天被老板骂了"
3. 小知回应："改了 5 遍还在改，说明你没放弃自己"
4. 金句卡片翻转：**"第 5 版不是失败，是第 5 次不肯妥协的自己"**
5. 用户截图保存，或者发给朋友
6. 第二天，ta 主动又打开了 App

## Executive Summary

### 产品愿景

Xiaozhi Journal 是一个提供情感陪伴的 AI 日记应用。大多数日记 App 做的是数据存储，真正稀缺的是情绪价值——用户需要的不是被记录，而是被理解。产品通过有温度的 AI 回应和情绪可视化，让用户在记录的同时"被看见"。

**产品一句话定义：** 别的日记应用关注记录，我们关注你的感受。

### 商业模式

Freemium（免费增值）：

| 层级 | 权益 |
|------|------|
| **免费版** | 每日有限次 AI 回应（基础额度），核心功能可用（打卡、波形图、历史回顾）|
| **付费版** | 无限 AI 回应、高级分析（周报/月报）、时间胶囊高级设置、优先客服 |
| **BYOK 模式** | 用户可自带 API Key，不限次数调用，仅支付平台基础服务费 |

### What Makes This Special

| 维度 | 竞品做法 | 我们的做法 |
|------|----------|-----------|
| AI 回应 | 冷冰冰的情绪分析 | 温暖的朋友式对话，有态度不评判 |
| 数据展示 | 线性记录列表 | 动态情绪波形图，一眼看见自己 |
| 用户价值 | 记录工具 | 情感陪伴 + 自我认知的镜子 |
| 启动成本 | 需要写一段文字 | 3 秒表情打卡，零门槛 |
| AI 成本 | 平台全包，用户无感知 | BYOK 模式让用户自行控制 AI 成本 |

**核心差异化：**
1. **"小知"AI 人格** — 温暖、有态度、不评判，回应像朋友不像客服
2. **今日金句** — 把碎碎念提炼成一句诗一样的话，天然可传播
3. **情绪波形图** — 7 天趋势一目了然，让用户"看见"自己
4. **时间胶囊** — 在最恰当的时机弹出历史共鸣，情感冲击极强
5. **BYOK 灵活模式** — 用户可自带 API Key，平台不限次，降低用户成本焦虑

**核心设计原则：**
- 不是记录工具，而是 **自我认知的镜子**
- 离线优先 — 无网络也可写日记、看波形图，数据先存本地，网络恢复自动同步
- 隐私优先 — 数据加密存储，用户可随时导出或删除
- AI 回应模式：先共情，再用独特角度帮用户重新看待事情，最后提炼金句

## AI 人格定义

**AI 的名字：** 小知

**人格特质：**
- **温暖但有个性** — 不是客服式的"加油，你可以的"，而是像朋友一样有态度
- **不评判** — 永远不说"你不应该这样想"，而是"我理解你为什么这样想"
- **敏锐** — 能捕捉用户没直接说出来的情绪
- **偶尔幽默** — 在合适的时机开个小玩笑

**回应风格示例：**

| 用户输入 | ❌ 差回应 | ✅ 好回应 |
|----------|----------|----------|
| "今天又被老板骂了" | "加油，下次会更好！" | "改了 5 遍还在改，说明你没放弃自己。方案可能不完美，但你在坚持。" |
| "感觉自己很没用" | "不要这样想，你很棒！" | "今天很难吧。但你能把这种感受说出来，本身就说明你在面对。" |
| "今天天气很好，开心！" | "开心是最好的情绪！" | "阳光好的时候心情也好，这种日子适合什么都不做，就发呆。" |

**系统 Prompt 核心：**
> 你是一个温暖、有个性的朋友。用户写日记后，你给出 2-3 句回应：先共情他的感受，再用一个独特的角度帮他重新看待这件事。不要用鸡汤话，要真实有共鸣。最后用一句话提炼这篇日记的"金句"，像一句诗。

**商业化约束：**
- 免费版和付费版的 AI 回应质量一致，不因付费等级降低回应质量
- 免费版仅限制每日调用次数，不限制回应质量
- BYOK 模式下用户自带 Key，平台不干预回应内容

## 首屏流程与 Onboarding 设计

### 冷启动状态（用户第一次打开）

1. **首屏：** 不是空白输入框，而是 — "今天心情怎么样？" + 5 个表情选项（😊 😐 😔 😡 😴）
2. **选择表情后：** 显示一个简短的引导语 — "随便写点什么吧，哪怕只有一句话"
3. **输入框出现：** 用户写完后点击"发送"
4. **AI 回应 + 金句卡片：** 打字机动画展示
5. **生成第一条情绪数据：** 波形图从这一点开始"生长"

### 已有数据状态（用户再次打开）

1. **首屏：** 7 天情绪波形图（核心视觉）
2. **下方：** "今天心情怎么样？" + 表情选择器
3. **点击任意表情：** 展开日记输入框

### Onboarding 流程（新用户注册后）

1. **欢迎页：** "欢迎，{昵称}。小知在这里陪你。"（3 秒自动跳过）
2. **AI 模式选择：** "你希望怎样使用 AI？" — 选项 A：平台提供 AI（每日免费 5 次）/ 选项 B：自带 API Key（不限次）
3. **首次引导：** 引导用户写第一篇日记，完成核心体验闭环
4. **完成：** 进入首页，波形图开始记录

## 空状态与 Fallback 设计

| 场景 | 处理方式 |
|------|----------|
| **没有日记数据** | 波形图显示引导文案："你的第一条日记从这里开始 ✨" |
| **API 调用超时** | 显示本地 fallback 回应："小知暂时不在，你的感受已经保存好了。稍后再来看看想对你说什么吧。" |
| **AI 回应质量差** | 在 prompt 中严格定义输出格式（JSON），如果解析失败则重试一次 |
| **网络断开** | 日记先保存到本地缓存，标记为 pending，网络恢复后异步同步到云端 |
| **BYOK Key 无效** | 温柔提示："你的 API Key 似乎不太对，检查一下？也可以先用平台 AI（今日剩余 {n} 次）" |

## User Journeys

### 旅程 1：注册/登录（新增）

**开场：** 用户通过朋友分享的金句卡片来到 Xiaozhi Journal，想自己试试。

**上升：** 看到简洁的登录页。用微信一键登录（或邮箱注册），3 秒完成。

**高潮：** 登录后看到 Onboarding："欢迎，{昵称}。小知在这里陪你。" 引导选择 AI 模式。

**结局：** 用户进入首页，看到引导波形图，准备写第一篇日记。

### 旅程 2：小林（忙碌职场人）— 核心成功路径

**开场：** 晚上 11 点，小林加班回家，脑子里一团乱。他打开 Xiaozhi Journal，不是想写日记，是想"把脑子里的东西倒出来"。

**上升：** 看到首页的波形图——过去 7 天有 5 天是"焦虑"和"疲惫"。他愣了一下，第一次直观地"看到"自己的状态。点了一个"😔"，输入框弹出来。

**高潮：** 他写了"今天又被老板骂了，方案改了 5 遍都不行"。3 秒后，小知回应："改了 5 遍还在改，说明你没放弃自己。方案可能不完美，但你在坚持。" 金句卡片翻转：**"第 5 版不是失败，是第 5 次不肯妥协的自己。"**

**结局：** 小林截了金句的图，发到朋友圈。第二天早上，他主动又打开 App 写了一篇日记。

### 旅程 3：小月（大学生）— 情感探索路径

**开场：** 小月有写手账的习惯，手机里已经装过 3 个日记 App，都死在"懒得打开"。朋友推荐了 Xiaozhi Journal。

**上升：** 打开 App，第一眼不是空白输入框，而是"今天心情怎么样？" + 5 个表情。她点了"😊"。引导语说"随便写点什么吧，哪怕只有一句话"。她写："今天论文通过了！导师说做得好！"

**高潮：** 小知说："开心的时候就该开心，不用谦虚。" 金句：**"原来一个小小的肯定，就能让人开心一整天。"** 然后弹窗提醒："3 个月前你也为论文焦虑过，但那次你也搞定了。"

**结局：** 小月觉得自己被"看见"了。她把金句分享给了闺蜜。一周后，她打开 App 看到了自己的情绪波形图——发现这周比上周开心多了。

### 旅程 4：阿琳（新手妈妈）— 碎片化场景路径

**开场：** 凌晨 2 点，宝宝终于睡了。阿琳一个人坐在客厅，心里很乱。她只有 30 秒。

**上升：** 打开 App，不想写长篇，点了一个"😔"。输入框弹出来，她打了三个字："好累啊。"

**高潮：** 小知回应："今天很难吧。但你能在凌晨 2 点还撑着，本身就很了不起。" 金句：**"凌晨 2 点的坚持，不需要任何人看见。"**

**结局：** 阿琳眼眶红了，但笑了。她把手机放下，心里轻了一点。第二天白天，她看到这条日记和金句，又笑了一下。

### 旅程 5：付费转化（新增）

**开场：** 用户免费使用了 2 周，每天写 1-2 篇日记，AI 每日额度快用完了。

**上升：** 提示条温和地出现："今天 AI 额度用完了。明天还有 5 次，或者升级到无限版，随时和小知聊天。"

**高潮：** 用户点击"看看有什么权益"，看到免费版 vs 付费版对比。被"周报/月报"和"无限 AI"打动。

**结局：** 用户完成支付，解锁无限 AI。第一次无限制地和小知对话。

### 旅程 6：错误恢复路径 — 网络断开场景

**开场：** 用户写了很长一篇日记，点击发送。但网络断了。

**上升：** 没有报错，没有崩溃。日记先保存到了本地缓存，显示："已保存。小知正在路上，网络恢复后就会回应你~"

**高潮：** 用户切出去干了别的，5 分钟后网络恢复，App 自动同步数据并调用 AI。

**结局：** 用户回来看到了温暖的 AI 回应。没有丢失任何数据，体验无缝。

## 多平台策略

### 平台覆盖

| 平台 | 技术 | 优先级 | 目标用户场景 |
|------|------|--------|-------------|
| **Web App** | Next.js | P0 | 桌面端深度使用、首次体验 |
| **微信小程序** | Taro | P1 | 碎片化记录、社交分享获客 |
| **移动 App** | Taro (RN/H5 壳) | P1 | 高频用户、推送通知、离线体验 |

### 功能差异化

| 功能 | Web | 小程序 | App |
|------|-----|--------|-----|
| 心情打卡 + 日记 | ✅ | ✅ | ✅ |
| AI 回应 + 金句 | ✅ | ✅ | ✅ |
| 情绪波形图 | ✅ | ✅ | ✅ |
| 时间胶囊 | ✅ | ✅ | ✅ |
| 金句分享 | ✅ 生成图片 | ✅ 微信分享卡片 | ✅ 系统分享 |
| 微信一键登录 | ✅ OAuth | ✅ 原生 | ✅ OAuth |
| 推送通知 | ❌（PWA 有限）| ✅ | ✅ |
| 离线能力 | ✅ IndexedDB + SW | ✅ 本地存储（200MB 限制）| ✅ AsyncStorage + SQLite |
| BYOK 配置 | ✅ | ✅ | ✅ |

### 本地缓存策略

系统采用**离线优先（Offline-First）**架构，所有平台均需支持本地缓存：

| 平台 | 缓存技术 | 能力 | 限制 |
|------|---------|------|------|
| **Web (Next.js)** | IndexedDB + Service Worker (PWA) | 离线读写、后台同步、缓存波形图渲染 | Service Worker 在 iOS Safari 支持有限 |
| **微信小程序** | wx.setStorageSync + 本地文件系统 | 同步缓存、文件缓存（图片/金句）| 存储上限 200MB；无后台同步 |
| **原生 App (Taro RN)** | AsyncStorage + SQLite | 离线读写、后台同步、大容量存储 | 需原生模块支持 SQLite |

**共享层：** 定义统一的 CacheProvider 接口（get/set/sync/delete），上层业务不关心具体存储实现。

**冲突解决：** 多端同时编辑时，以最后写入时间为准，用户不会丢失任何日记。

## Functional Requirements

### 用户认证

**邮箱注册与确认（FR1）**

- FR1.1: 用户注册后必须验证邮箱才能登录
- FR1.2: 系统提供两种邮箱确认方式供用户选择：
  - 链接确认：发送确认链接，点击即验证
  - 验证码确认：发送 6 位数字验证码，用户输入验证（自定义 OTP）
- FR1.3: 验证码/确认链接有效期 10 分钟，过期需重新申请
- FR1.4: 密码长度至少 8 位（前后端一致验证）
- FR1.5: 密码必须包含大小写字母和数字（前后端一致验证）
- FR1.6: 未验证邮箱的账户 7 天后自动清理

**第三方登录（FR2）**

- FR2: 用户可通过微信一键登录（OAuth 2.0）

**密码管理（FR3）**

- FR3.1: 用户可重置密码（邮箱验证链接）
- FR3.2: 重置链接有效期 1 小时，过期需重新申请
- FR3.3: 修改密码需重新认证（secure_password_change = true）

**个人资料与安全审计（FR4）**

- FR4.1: 用户可管理个人资料（昵称、头像）
- FR4.2: 系统记录登录日志（时间、IP、设备信息）
- FR4.3: 系统记录安全事件（密码修改、邮箱变更、API Key 操作）
- FR4.4: 用户可查看最近 30 天登录历史

### 日记记录

- FR5: 用户可以选择心情表情（😊 😐 😔 😡 😴）来记录当天情绪
- FR6: 用户可以输入自由文本来记录日记内容
- FR7: 用户可以保存日记条目
- FR8: 用户可以查看自己的历史日记列表（支持分页）
- FR9: 用户可以查看单条日记的详情（内容 + AI 回应 + 金句）

### AI 互动

- FR10: 用户在保存日记后可以获得 AI 的回应
- FR11: 用户可以收到 AI 提炼的"今日金句"
- FR12: 用户可以获得 AI 基于最近 7 天情绪数据生成的趋势总结
- FR13: 用户可以在 AI 回应失败时看到清晰的中文降级提示，不影响已保存的日记内容
- FR14: 用户可配置自己的 API Key（BYOK），用于替代平台 AI 调用

### AI 额度管理

- FR15: 免费用户每日可使用有限次平台 AI（默认 5 次/天）
- FR16: 付费用户无限次使用平台 AI
- FR17: BYOK 用户不限次数，使用自己的 API Key 调用
- FR18: 用户可查看当日剩余 AI 额度
- FR19: 额度用尽时，系统温和引导用户升级或配置 BYOK

### 情绪可视化

- FR20: 用户可以查看最近 7 天的情绪趋势波形图
- FR21: 用户可以在波形图上看到每条记录对应的心情表情
- FR22: 用户在没有数据时可以看到引导性空状态

### 时间胶囊

- FR23: 用户可以收到系统推送的多时间窗口梯度日记提醒（周年/半年/季度），标题根据实际时间差动态生成
- FR24: 用户可以点击查看被推送的历史日记
- FR25: 系统对时间胶囊进行频率控制（24h 不重复、同一篇 30 天冷却）

### 数据管理

- FR26: 用户的日记数据在云端数据库中持久化存储，跨设备同步
- FR27: 用户的日记在网络断开时仍可保存到本地缓存，网络恢复后自动同步到云端
- FR28: 多端同时编辑时，系统以最后写入时间为准解决冲突
- FR29: 用户可随时导出自己的全部日记数据
- FR30: 用户可随时删除自己的账户和全部数据

### 分享

- FR31: 用户可以将金句生成为可分享的卡片图片
- FR32: 用户可通过微信分享金句卡片（小程序端原生分享，Web 端图片分享）

### 付费与订阅

- FR33: 用户可查看免费版与付费版的权益对比
- FR34: 用户可通过微信支付或支付宝完成订阅
- FR35: 用户可随时取消订阅，已付费权益持续到当前周期结束

## Non-Functional Requirements

### 性能

- NFR1: Web 首屏加载时间 ≤ 2 秒（用户打开到看到波形图和心情选择器）
- NFR2: 小程序首屏加载时间 ≤ 1.5 秒
- NFR3: AI 回应 P95 响应时间 ≤ 3s，P99 ≤ 8s
- NFR4: 情绪波形图渲染 ≤ 500ms
- NFR5: 多端数据同步延迟 ≤ 3 秒（网络正常）

### 安全

**传输加密（NFR6）**

- NFR6.1: 所有用户数据在传输中使用 TLS 1.2+ 加密
- NFR6.2: SSL 证书有效期 ≥ 90 天，自动续期
- NFR6.3: 验证方式：通过安全审计/渗透测试验证，HTTPS 强制跳转

**敏感数据存储（NFR7）**

- NFR7.1: 用户 BYOK API Key 使用 AES-256-GCM 加密存储
- NFR7.2: 密钥派生使用 scrypt（N=2^14, r=8, p=1），含 Salt
- NFR7.3: 验证方式：通过安全审计验证，密钥与加密数据分离存储

**平台 API Key 保护（NFR8）**

- NFR8.1: 平台 AI API Key 仅在服务端 Route Handler 使用
- NFR8.2: API Key 存储于环境变量，不提交 Git，不暴露客户端
- NFR8.3: 验证方式：通过代码审计验证，无客户端直接调用 AI API

**密码哈希（NFR9）**

- NFR9.1: 用户密码使用 bcrypt 哈希存储，成本因子 ≥ 10
- NFR9.2: 验证方式：通过安全审计验证 bcrypt 配置

**行级安全（NFR10）**

- NFR10.1: RLS 策略确保用户只能访问自己的数据
- NFR10.2: 所有业务表（journals, profiles, ai_usage, subscriptions, user_api_keys）启用 RLS
- NFR10.3: 验证方式：通过 SQL 注入测试验证，匿名用户无法查询数据

**Session 管理（NFR27-NFR29）**

- NFR27: JWT 过期时间 1 小时，Refresh token 自动轮换
- NFR28: Session 超时：7 天不活动自动登出，最大生命周期 30 天
- NFR29: 登录速率限制：5 分钟内最多 10 次登录尝试，超过需验证码

### 可用性

- NFR11: 系统月度可用性 ≥ 99.5%（SLA）
- NFR12: 离线状态下用户可写日记、看波形图、回顾历史，数据不丢失
- NFR13: 网络恢复后本地缓存数据自动同步到云端，无需用户手动操作

### 可扩展性

- NFR14: 系统支持从 1K DAU 增长到 10K DAU 无需架构变更（通过云服务自动扩展）
- NFR15: AI 调用成本随用户增长线性可控，不因并发增加而失控

### 合规

- NFR16: 用户可随时导出自己的全部数据（GDPR 数据可携带权）
- NFR17: 用户删除账户后，数据在 30 天内彻底清除（GDPR 被遗忘权）
- NFR18: 应用提供隐私政策页面，说明数据收集、存储、使用方式
- NFR19: 危机词检测不做医疗诊断，仅提供温柔引导和专业帮助资源链接
- NFR22: 情绪日记数据分类为敏感健康数据，适用更严格的存储和访问保护标准
- NFR23: 用户注册时确认适用年龄（18+）或需要监护人同意

### 成本可控

- NFR20: 平台 AI 调用成本不超过总收入的 40%
- NFR21: 系统提供 AI 调用成本看板（按日/周/月统计）
