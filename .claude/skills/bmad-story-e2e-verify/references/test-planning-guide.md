# 测试规划指南

## 从 Story 生成测试计划

每个 Story 文件包含：
- **验收标准（Acceptance Criteria）** → 直接映射为 Happy Path 测试用例
- **技术实现细节（Implementation Notes）** → 提示需要覆盖的技术路径
- **边缘情况（Edge Cases）** → 从 bmad-review-edge-case-hunter 的输出中提取

### 测试用例分类

| 类型 | 来源 | 示例 |
|------|------|------|
| Happy Path | Story AC | "用户输入邮箱+密码，点击注册，跳转成功" |
| 错误路径 | Story AC + 常识 | "邮箱格式错误时显示提示" |
| 边缘情况 | review 输出 | "断网后重新连接，数据是否正确同步" |
| 回归检查 | 已有功能 | "注册后刷新页面，登录状态是否保持" |

### 测试计划结构

```markdown
## 测试计划: [Story ID] - [Story Title]

### 路由覆盖
- [ ] GET /route-1 — 页面渲染
- [ ] GET /route-2 — 页面渲染

### Happy Path
1. [TC-01] 描述 → 步骤 → 预期结果

### 错误路径
2. [TC-02] 描述 → 步骤 → 预期结果

### 边缘情况
3. [TC-03] 描述 → 步骤 → 预期结果

### 回归检查
4. [TC-04] 描述 → 步骤 → 预期结果
```

## 覆盖度评估

测试计划完成后，自检：
- 是否覆盖了 Story AC 的每一条？
- 是否覆盖了 Story 涉及的所有路由？
- 是否包含了至少 2 个边缘情况？
- 是否包含了至少 1 个回归检查？

如果任一答案为否，补充测试用例后再进入执行阶段。
